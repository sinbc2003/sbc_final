## 1. 최종 목표

**구형 노트북을 가진 교사들도 로컬 LLM으로 문서 자동화(TeacherFlow)를 쓰게 한다.** API(하이엔드 LLM) 연결도 옵션으로 유지.
성공 기준(사용자 정의): `"C:\Users\sinbc\OneDrive\바탕 화면\2024학년도 기간제 교사 채용 점수 집계표(역사, 체육, 러시아어, 프랑스어).hwpx"` 같은 병합 표 문서를 **gemma4 (2B)** 가 채우면 성공. 이 문서는 예시일 뿐이고 더 복잡한 표 문서들이 실전 대상.

## 2. 핵심 결론 (합의된 판단)

1. **가능하다.** 단, 모델을 키우는 문제가 아니라 **시스템(스캔·주소지정·검증)을 견고하게 만드는 문제**다.
2. 설계 대원칙: **"모델이 작아질수록 지능을 코드로 옮긴다."**
   - 위치 계산·산술(순위/평균)·HWPX 생성·검증 = 파이썬 (결정적)
   - 의미 매칭("이 값이 어느 라벨에 가나") = LLM (작은 모델도 가능한 유일한 판단)
   - 출력 형식 = llama-server GBNF/JSON 스키마 강제 (틀리고 싶어도 못 틀리게)
   - 결과 = 재추출 자동 검증 + 재시도 (작은 모델의 실수를 시스템이 흡수)
3. 작업 난이도 분류:
   - **쉬움 (LoRA 불필요, 즉시 가능)**: 표 빈칸 채우기·원본 직접 수정, 분류, 추출, 짧은 요약 — JSON 문법 강제만 붙이면 2B로 충분
   - **어려움 (LoRA 필요)**: 장문 문서 생성(가정통신문·계획서를 처음부터) — 증류 LoRA + A4 10~15p는 "개요 생성 → 절 단위 생성" 분할 파이프라인 필수 (한 방 생성은 컨텍스트·일관성 모두 불가)
4. Inline AI가 복잡한 표를 잘 채우는 것은 LLM이 천재라서가 아니라 **결정적 스캔·blockId 시스템이 떠먹여 주기 때문** (역공학으로 확증, §6). 같은 시스템을 만들면 그 자리에 gemma를 꽂을 수 있다.
